**Feature ID**: `rpi-emergency-failover`
**작성일**: 2026-05-20
**작성자**: bkit PDCA (sadojs@gmail.com)
**대상 하드웨어**: Raspberry Pi 3B (1GB RAM, ARMv7, Cortex-A53 1.2GHz)
**상태**: Plan (Review)

## 1. 배경 및 문제 정의

### 현재 아키텍처 요약
- **서버 측 (NestJS Backend)**: 자동화 룰 엔진(`automation-runner.service.ts` 950라인), 관수 스케줄러(`irrigation-scheduler.service.ts` 443라인), MQTT 브로커, DB, WebSocket
- **라즈베리파이 측**: `config-agent` (설정 배포), `gpio-agent` (릴레이 제어), `zigbee2mqtt` (Zigbee 게이트웨이), `mosquitto` (로컬 MQTT — 옵션)
- **통신 흐름**: 자동화 룰 평가 → 서버에서 의사결정 → MQTT 명령 발행 → RPi가 GPIO/Zigbee로 장치 제어

### 핵심 위험 시나리오
1. **서버↔RPi 단절 시 장치 ON 상태 고착**
   - 관수 ON 후 OFF 신호 못 받음 → 침수 / 작물 침해
   - 환기/개폐기 닫힘 상태에서 고온 → 작물 고사
   - 비 감지 후 개폐기 닫기 명령 미도달 → 누수 피해
2. **인터넷 회선 불안정**: 농촌 지역 LTE/광랜 단절은 분 단위~시간 단위 발생 가능
3. **중앙 서버 장애**: 단일 서버에 모든 농장이 의존 → SPOF (Single Point of Failure)

## 2. 3가지 옵션 검토

### 옵션 1: 현행 유지 (Server-Centric)

**구조**
```
[Server] ←─MQTT─→ [RPi: gpio-agent + z2m] ←→ [Zigbee/GPIO 장치]
   ↑
   모든 자동화 룰 실행
```

**가능 여부**: ✅ 현재 구현 완료 상태 (안정 운영 중)

**장점**
- 구현 부담 0 (현 상태 유지)
- 자동화 룰 변경 시 즉시 반영
- 중앙 집중식 모니터링/로그 일원화
- RPi 펌웨어 업그레이드 불필요

**단점 (치명적)**
- ❌ **단일 장애점 (SPOF)**: 서버 또는 네트워크 단절 시 자동화 완전 중단
- ❌ **장치 OFF 신호 미도달 위험**: 관수 침수, 비 닫힘 실패
- ❌ **고온 시 환기 미동작** → 작물 손실 (배상/신뢰도 직격)
- ❌ **농장주 SLA 보장 불가**: "왜 자동으로 안 꺼졌냐"는 클레임 대응 불가

**결론**: 이머전시 플랜으로 **부적합**. 현재의 위험을 해소하지 않음.

### 옵션 2: 서버 우선 + RPi 기본 룰 폴백 (Hybrid Fallback)

**구조**
```
정상:    [Server] ──룰──→ [RPi: gpio-agent + fallback-engine] → 장치
단절 시: [RPi: fallback-engine (시간/온도 기반 기본 룰)] → 장치
```

**RPi에 추가될 기본 룰 예시**
- 개폐기: 07:00 OPEN, 19:00 CLOSE (시간 기반)
- 관수: 08:00부터 30분간 ON (시간 기반)
- 환기: 내부 온도 > 35°C → 환기팬 ON, < 28°C → OFF (온도 임계값)
- 비 감지: 빗물 센서 ACTIVE → 모든 개폐기 CLOSE (안전 우선)
- **워치독**: 서버 하트비트 N분(예: 3분) 미수신 → 폴백 모드 진입

**가능 여부**: ✅ **구현 가능** (RPi 3B 성능으로 충분)

**필요 작업량** (사용자 사양 반영 후 재추정)
| 항목 | 작업 시간 | 비고 |
|------|-----------|------|
| RPi `fallback-engine` 모듈 신규 | 4~6일 | Node.js, ~700라인 (월별 스케줄러 포함) |
| 서버 하트비트 발행 | 0.5일 | `farm/{id}/server/heartbeat` (10초 주기) |
| RPi 워치독 + 모드 전환 | 1일 | 진입(3분)/이탈(30초 grace) |
| 폴백 룰 DB 스키마 + 동기화 (서버↔RPi) | 2.5일 | `fallback_rules` 테이블, 양방향 sync |
| 빗물 센서 override 통합 | 0.5일 | 기존 rain-override와 연동 |
| 폴백 로그 → 복구 후 서버 전송 | 1일 | 로컬 SQLite 큐 |
| UI: 월별 개폐기 스케줄 편집 화면 (+ 버튼) | 2.5일 | 12개월 카드, 활성/비활성, always-open 모드 |
| UI: 관수/액비/환기팬/유동팬 폴백 설정 | 1일 | 토글 + 임계값 입력 |
| E2E 테스트 (서버 다운, 단절 시나리오 5종) | 1.5일 | 월 경계, 빗물 override, 30분 타임아웃 등 |
| **합계** | **약 14.5~17일** | |

**장점**
- ✅ 자동화 룰의 풍부함은 서버에서 유지 (변경 빈도 높은 룰은 서버 측)
- ✅ 단절 시에도 최소한의 안전 동작 보장 (관수 차단, 개폐기 시간 제어)
- ✅ 변경 부담이 옵션 3보다 훨씬 작음
- ✅ RPi 3B로도 시간/온도 기반 단순 룰은 가뿐히 처리 가능 (CPU/메모리 여유)

**단점**
- ⚠️ **이중 룰 관리**: 서버 룰과 RPi 폴백 룰의 동기화 필요 (혼동 위험)
- ⚠️ 폴백 진입/이탈 경계가 모호하면 "이중 실행" 발생 가능 → MQTT QoS, idempotent 보장 설계 필요
- ⚠️ 폴백 룰이 너무 단순하면 단절 장기화 시 작물 손실 여전 (예: 한겨울 폴백 룰이 여름용이면 위험)

**결론**: ⭐ **실용적 균형점**. 비용 대비 효용 최고.

### 옵션 3: RPi 우선 (Edge-Centric, 서버는 UI/데이터 표시만)

**구조**
```
[Server: UI + 데이터 표시 + 로그 집계]
   ↑ (데이터 동기화만)
   ↓
[RPi: 자동화 엔진 풀버전 + 룰 DB + 스케줄러 + GPIO/Zigbee]
```

서버는 **명령을 내리지 않고**, RPi가 자율적으로 자동화 실행. 농장주가 웹에서 설정 변경 → RPi에 동기화 → RPi가 실행.

**가능 여부**: ⚠️ **기술적으로는 가능하나, RPi 3B 사양과 작업량 측면에서 매우 어려움**

**RPi 3B 사양 검토**
| 항목 | RPi 3B | 현재 사용 중 | 자동화 엔진 추가 시 |
|------|--------|--------------|---------------------|
| CPU | Cortex-A53 4코어 1.2GHz | z2m + mosquitto + agents | + 룰 엔진 + 스케줄러 |
| RAM | 1GB | ~500MB 사용 추정 | +200~300MB 예상 (Node + SQLite) |
| 저장소 | SD 카드 (I/O 느림) | OK | DB 쓰기 빈번 → SD 수명 저하 위험 |
| 네트워크 | 100Mbps + Wi-Fi | OK | OK |

→ **메모리 한계가 가장 큰 리스크**. z2m이 200~400MB 사용 + 자동화 엔진 + Node 런타임 → 1GB 부족 가능성. RPi 4 (2GB+) 권장하나, 사용자는 **3B 명시**.

**필요 작업량** (예상)
| 항목 | 작업 시간 | 비고 |
|------|-----------|------|
| 백엔드 automation 모듈 RPi 포팅 (~2000라인) | 15~20일 | TypeScript → Node, TypeORM → SQLite |
| RPi 로컬 DB (SQLite) 스키마 + 마이그레이션 | 3일 | |
| 서버 ↔ RPi 양방향 설정 동기화 | 5일 | conflict resolution 포함 |
| Zigbee2MQTT 룰 평가용 데이터 인터페이스 | 3일 | |
| 다중 사용자 RBAC를 서버에서 RPi로 위임 | 3일 | "농장주 인증" 흐름 재설계 |
| RPi 자동화 엔진 메모리 최적화 | 5일 | RPi 3B 1GB 한계 대응 |
| 룰 충돌 / 중복 실행 방지 (서버 명령 비활성화) | 2일 | |
| 전체 UI 데이터 흐름 재설계 | 5일 | 모든 화면이 "RPi의 미러" |
| E2E 테스트 + 마이그레이션 가이드 | 5일 | |
| **합계** | **약 46~51일 (1~2개월)** | |

**장점**
- ✅ 완전한 자율 동작 (서버 영구 단절돼도 자동화 그대로)
- ✅ SPOF 완전 제거
- ✅ 농장 단위 격리 → 한 농장 문제가 다른 농장에 영향 없음

**단점 (심각)**
- ❌ **작업량 폭증** (옵션 2의 약 4배)
- ❌ **RPi 3B 메모리 한계**로 안정성 의심 → RPi 4/5 강제 업그레이드 필요할 가능성 큼
- ❌ **이중 시스템 유지보수**: 서버 + RPi 양쪽에 자동화 코드 → 버그 픽스/기능 추가 시 두 곳 수정
- ❌ **현재 코드베이스 대규모 재작성** → 기존 안정성 무너질 위험
- ❌ **다중 농장주 시나리오 복잡**: 같은 RPi에 농장주 2명 접속 시 권한/세션 관리 어려움
- ❌ **SD 카드 쓰기 빈번** → SD 수명 단축 (현장 대량 운영 시 교체 비용 증가)
- ❌ **OTA 업데이트 위험**: RPi의 자동화 엔진 버그 패치는 전체 농장 OTA 필요 → 회수/장애 리스크 큼

**결론**: ❌ **현 시점에서는 비현실적**. RPi 3B에서 무리. 장기 비전으로는 가능하나 RPi 4 이상 + 충분한 개발 기간 필수.

## 4. 추천: **옵션 2 (Hybrid Fallback)**

### 추천 사유
1. **위험 대비 비용 최적**: 가장 큰 위험(관수 침수, 환기 실패)을 10~13일 작업으로 해소
2. **RPi 3B 사양으로 충분**: 시간/온도 임계값 기반 단순 룰은 CPU/RAM 부담 미미
3. **현 아키텍처 유지하면서 안전망만 추가**: 기존 코드 안정성 보존
4. **점진적 확장 가능**: 향후 옵션 3로 진화하고 싶으면, 폴백 엔진을 점차 확장하면 됨 (점진 이행 경로 확보)
5. **농장주 신뢰도 즉시 향상**: "단절돼도 안전합니다" 마케팅/SLA 가능

### 폴백 룰 사양 (사용자 확정 — 2026-05-20)

모든 폴백 룰은 **환경 설정 파일 + 웹 UI**에서 편집 가능. RPi에 로컬 저장(`/etc/smartfarm/fallback-rules.json`) + 서버 DB 미러링 (`fallback_rules` 테이블).

#### 1) 개폐기 (천창/측창) — 월별 스케줄

| 월 | 기본값 | 비고 |
|---|--------|------|
| 1월 | (비활성) | + 버튼으로 추가 가능 |
| 2월 | (비활성) | + 버튼으로 추가 가능 |
| 3월 | (비활성) | + 버튼으로 추가 가능 |
| **4월** | 09:00 OPEN / 17:00 CLOSE | 기본 활성 |
| **5월** | 08:00 OPEN / 18:00 CLOSE | 기본 활성 |
| **6월** | (24시간 OPEN 유지) | 폴백 동작 없음 (여름 환기) |
| **7월** | (24시간 OPEN 유지) | 폴백 동작 없음 |
| **8월** | (24시간 OPEN 유지) | 폴백 동작 없음 |
| **9월** | (24시간 OPEN 유지) | 폴백 동작 없음 |
| **10월** | 08:00 OPEN / 18:00 CLOSE | 기본 활성 |
| **11월** | 09:00 OPEN / 17:00 CLOSE | 기본 **비활성** (값은 저장, 토글로 활성화) |
| 12월 | (비활성) | + 버튼으로 추가 가능 |

**UI 요구사항**
- 월별 카드 12개 표시 (활성/비활성 토글)
- "+ 월 추가" 버튼으로 비활성 월 활성화
- 각 월: OPEN 시각 / CLOSE 시각 / 24h-OPEN 모드(체크박스)
- 빗물 센서 override: ACTIVE → 즉시 CLOSE, 비 종료 → 직전 스케줄 상태로 복귀 (OPEN 시간대면 OPEN, CLOSE 시간대면 CLOSE)

#### 2) 관수 (Irrigation)

| 동작 | 사양 |
|------|------|
| 통신 단절 감지 시 | **관수 시작 시점부터 30분 경과하면 자동 OFF** (단절 시점이 아닌 ON 시점 기준) |
| 폴백 중 신규 ON | 금지 (단절 중에는 관수 시작 안 함) |
| 통신 복구 시 | 서버 룰에 권한 위임 (즉시 OFF 해제는 안 함) |

> 30분은 환경 설정에서 조정 가능 (`irrigation.maxRuntimeMinutes`, 기본 30).
> **카운트 기준**: 관수 채널이 ON 된 timestamp 기준. RPi가 채널별 `onSince` 시각을 보관하다 폴백 진입 시점에 `now - onSince >= 30min` 인 채널은 즉시 OFF, 그 외는 `onSince + 30min` 시각에 OFF.

#### 3) 액비 / 양액기 (Liquid Fertilizer)

| 동작 | 사양 |
|------|------|
| 통신 단절 감지 시 | **즉시 OFF (지연 없음)** |
| 폴백 중 | 무조건 OFF 유지 |
| 통신 복구 시 | 서버 룰에 권한 위임 |

#### 4) 환기팬 / 유동팬 (Ventilation Fan)

> **참고**: 본 프로젝트에서 "환기팬"과 "유동팬"은 동일한 장치를 가리킴. 단일 카테고리로 통합 관리.

| 동작 | 사양 |
|------|------|
| 기본 활성 상태 | ❌ **비활성 (default disabled)** |
| 활성 시 ON 조건 | 내부 온도 > **온도ON 임계값** (기본 35°C, 편집 가능) |
| 활성 시 OFF 조건 | 내부 온도 < **온도OFF 임계값** (기본 28°C, 편집 가능) |
| 온도 임계값 | UI 또는 설정 파일에서 조정 가능 (`fan.onTemp`, `fan.offTemp`) |
| 유효성 검증 | `onTemp > offTemp` 강제 (그렇지 않으면 저장 차단) |

> 사용자가 UI에서 명시적으로 "환기팬 폴백 활성화" 토글을 켜야 동작.

#### 5) 기타 장치 (default deny)

명시되지 않은 모든 액추에이터는 **단절 감지 시 OFF**. 폴백 룰 매트릭스에 등록된 채널만 위 정책 적용.

### 폴백 진입/이탈 정책

- **진입**: 서버 하트비트 **5분 미수신** → 폴백 모드 ON, MQTT 토픽 `farm/{id}/mode` = `fallback` 발행
- **이탈**: 서버 하트비트 재수신 + 30초 안정화 → 폴백 모드 OFF, `mode` = `online`
- **이중 실행 방지**: 폴백 모드 중에는 서버 명령(`farm/{id}/gpio/relay`) 무시. 단, 비상 정지 명령(`farm/{id}/gpio/emergency-stop`)은 항상 허용
- **빗물 센서**: 폴백/정상 모드 무관 → 항상 최우선 (개폐기 강제 CLOSE)
